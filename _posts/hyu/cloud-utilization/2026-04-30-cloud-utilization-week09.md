# 클라우드활용 9주차: AWS 아키텍처 기초 설계

이번 주차의 핵심은 AWS 서비스를 하나씩 외우는 것이 아니라, 여러 서비스를 어떤 구조로 연결해야 하나의 시스템이 되는지 이해하는 것이다. 클라우드 아키텍처는 서버, 데이터베이스, 스토리지, 네트워크, 보안 요소를 배치하고 연결하는 설계도에 가깝다.

## 아키텍처

아키텍처는 시스템을 구성하는 컴포넌트와 그들 사이의 관계, 데이터 흐름, 트래픽 흐름을 정의하는 설계이다.

건축에서 설계도가 건물의 구조를 잡는 것처럼, 클라우드 아키텍처는 서비스가 어떤 방식으로 연결되고 동작해야 하는지를 정리한다. 아키텍처를 명확히 잡지 않으면 시스템이 커질수록 유지보수 비용이 증가하고 장애 대응도 어려워진다.

## 좋은 아키텍처의 조건

좋은 아키텍처는 단순히 동작하는 구조가 아니라, 안정적으로 운영되고 필요할 때 확장되며 장애 상황에서도 복구 가능한 구조이다.

AWS에서는 좋은 아키텍처를 이해할 때 다음 여섯 가지 관점을 중요하게 본다.

| 관점 | 의미 |
|---|---|
| 운영 우수성 | 시스템을 안정적으로 운영하고 개선할 수 있는 능력 |
| 보안성 | 데이터 보호와 접근 제어를 고려한 구조 |
| 신뢰성 | 장애가 발생해도 복구 가능한 구조 |
| 성능 효율성 | 필요한 성능을 자원 낭비 없이 제공하는 능력 |
| 비용 최적화 | 필요한 만큼만 자원을 사용하도록 설계하는 것 |
| 지속 가능성 | 클라우드 워크로드 운영 시 환경적 영향을 줄이는 관점 |

## 운영 우수성

운영 우수성은 시스템을 운영하면서 문제를 빠르게 발견하고 개선할 수 있는 능력이다.

단순히 장애가 발생했을 때 복구하는 것만 의미하지 않는다. 운영 매뉴얼, Runbook, Playbook, 자동화, 로그, 메트릭, 추적 정보 등을 통해 시스템 상태를 계속 파악하고 개선하는 것이 핵심이다.

## 보안성

보안성은 데이터와 시스템을 보호하고, 필요한 사용자에게만 필요한 권한을 주는 구조이다.

클라우드에서는 보안이 물리 장비를 잠그는 문제로 끝나지 않는다. IAM 권한, 네트워크 접근 제어, 보안 그룹, 암호화, 로그 추적, WAF, GuardDuty 같은 서비스가 함께 연결된다.

## 신뢰성

신뢰성은 장애가 발생해도 시스템이 계속 동작하거나 빠르게 복구되는 능력이다.

클라우드에서 신뢰성을 높이려면 하나의 서버나 하나의 네트워크 구간에 모든 기능이 몰리지 않게 해야 한다. 여러 가용 영역에 리소스를 분산하고, 장애 발생 시 다른 리소스로 전환될 수 있게 설계해야 한다.

## 성능 효율성

성능 효율성은 필요한 처리 속도를 제공하면서도 자원을 낭비하지 않는 구조이다.

성능이 부족하면 사용자 경험이 나빠지고, 과도하게 큰 자원을 사용하면 비용이 낭비된다. 따라서 시스템 요구사항에 맞는 컴퓨팅, 스토리지, 데이터베이스, 네트워크 구성을 선택해야 한다.

## 비용 최적화

비용 최적화는 단순히 가장 저렴한 서비스를 선택하는 것이 아니다.

필요한 만큼만 자원을 사용하고, 사용하지 않는 리소스를 줄이며, 서비스 특성에 맞는 과금 모델을 선택하는 것이다. 클라우드는 사용량 기반 과금이기 때문에 설계가 곧 비용과 연결된다.

## 지속 가능성

지속 가능성은 클라우드 워크로드를 운영할 때 환경적 영향을 줄이는 관점이다.

불필요한 자원 사용을 줄이고, 효율적인 리소스 선택과 자동 확장을 통해 낭비를 줄이는 것도 클라우드 아키텍처의 일부로 볼 수 있다.

## AWS 아키텍처 기본 요소

AWS 아키텍처는 여러 종류의 서비스가 함께 구성된다.

| 요소 | 역할 | 대표 서비스 |
|---|---|---|
| 컴퓨팅 | 애플리케이션 실행 | EC2, Lambda |
| 스토리지 | 파일과 객체 저장 | S3, EBS, EFS |
| 네트워크 | 리소스 연결과 트래픽 제어 | VPC, Load Balancer, Route 53 |
| 데이터베이스 | 구조화된 데이터 저장 | RDS, DynamoDB |
| 보안 | 접근 제어와 데이터 보호 | IAM, WAF, GuardDuty |
| 모니터링 | 상태 추적과 로그 기록 | CloudWatch, CloudTrail |

## VPC 기반 구성

VPC는 AWS 리소스를 담는 가상 네트워크이다.

서브넷은 VPC 안에서 네트워크 영역을 나누는 단위이고, 퍼블릭 서브넷은 인터넷과 연결되는 영역, 프라이빗 서브넷은 외부에서 직접 접근하지 않는 내부 영역으로 볼 수 있다.

인터넷 게이트웨이는 VPC가 외부 인터넷과 통신할 수 있게 해주고, 라우팅 테이블은 트래픽이 어디로 이동해야 하는지 결정한다. 보안 그룹은 인스턴스 단위의 가상 방화벽 역할을 한다.

## 아키텍처 다이어그램

아키텍처 다이어그램은 클라우드 구조를 그림으로 표현한 것이다.

사용자, 리전, 가용 영역, VPC, 서브넷, 인터넷 게이트웨이, EC2, S3, RDS 같은 요소를 배치하고 요청 흐름을 화살표로 표현하면 서비스가 어떤 방식으로 동작하는지 한눈에 볼 수 있다.

## 한 줄 정리

클라우드 아키텍처는 AWS 서비스를 따로따로 쓰는 것이 아니라, 컴퓨팅·스토리지·네트워크·보안·모니터링 요소를 목적에 맞게 연결하는 설계이다.
