# 클라우드활용 1주차: AWS 클라우드 컴퓨팅 개요와 글로벌 인프라 구조

## 1. 이번 주차에서 다룬 내용

이번 주차에서는 클라우드 컴퓨팅의 기본 개념에서 시작해 AWS의 주요 서비스와 글로벌 인프라 구조까지 훑었다. 이 주차는 이후 EC2, S3, IAM, VPC 같은 개별 서비스를 배우기 위한 기반이다. 중요한 것은 AWS 서비스 이름을 많이 외우는 것이 아니라, 클라우드가 기존 온프레미스 운영 방식과 무엇이 다른지 이해하는 것이다.

이번 주차에서 다룬 내용은 다음과 같다.

- 클라우드 컴퓨팅의 개념과 필요성
- 온프레미스 환경과 클라우드 환경의 차이
- 클라우드 도입 시 기대 효과
- AWS의 탄생 배경과 주요 서비스 영역
- 리전, 가용 영역, 엣지 로케이션 등 AWS 글로벌 인프라 구조
- 리전 선택 시 고려해야 하는 요소

## 2. Background

이번 주차를 이해하기 위해 필요한 배경지식은 다음과 같다.

- IT 리소스:
  서버, 스토리지, 네트워크, 데이터베이스처럼 서비스를 운영하기 위해 필요한 컴퓨팅 자원이다.

- 온프레미스:
  기업이나 기관이 자체 데이터센터 또는 서버실에 장비를 두고 직접 운영하는 방식이다.

- 온디맨드:
  필요한 순간에 필요한 만큼 자원을 요청해서 사용하는 방식이다.

- 종량제 과금:
  리소스를 구매하는 것이 아니라 사용한 만큼 비용을 지불하는 방식이다.

- 고가용성:
  장애가 발생해도 서비스가 계속 동작할 수 있도록 설계하는 성질이다.

### 보충 설명

클라우드를 처음 배울 때 “결국 남의 컴퓨터를 빌려 쓰는 것 아닌가?”라고 생각할 수 있다. 큰 방향에서는 맞지만, 그 정도로만 이해하면 부족하다. 클라우드의 핵심은 서버의 위치가 바뀌는 것이 아니라, 인프라를 소비하고 운영하는 방식이 바뀌는 것이다.

온프레미스에서는 서버 구매, 설치, 전원, 네트워크, 장애 대응, 장비 교체까지 직접 관리해야 한다. 반면 클라우드에서는 필요한 리소스를 서비스처럼 생성하고, 사용량에 따라 비용을 지불하며, 필요 없으면 제거할 수 있다.

## 3. 개념 정리

### 3.1 클라우드 컴퓨팅

클라우드 컴퓨팅은 IT 리소스를 인터넷을 통해 온디맨드로 제공받고, 사용한 만큼 비용을 지불하는 방식이다.

내가 이해한 방식으로 정리하면 클라우드는 “서버를 사는 방식”에서 “서버 사용 능력을 빌리는 방식”으로 넘어가는 것이다. 서버를 직접 구매하면 처음부터 큰 비용과 시간이 필요하지만, 클라우드에서는 콘솔이나 CLI를 통해 필요한 리소스를 빠르게 만들 수 있다.

이 차이는 개발 속도에도 영향을 준다. 예전에는 서버가 준비될 때까지 기다려야 했지만, 클라우드에서는 개발자가 직접 테스트 서버를 만들고 삭제할 수 있다.

### 3.2 온프레미스와 클라우드 비교

| 구분 | 온프레미스 | 클라우드 |
|---|---|---|
| 인프라 소유 | 기업이 직접 소유 | 클라우드 제공자의 인프라 사용 |
| 초기 비용 | 서버, 장비 구매 비용이 큼 | 초기 비용이 낮고 사용량 기반 과금 |
| 확장 방식 | 장비 추가 구매와 설치 필요 | 필요 시 리소스 확장 가능 |
| 유지보수 | 직접 패치, 교체, 장애 대응 | 일부 영역은 AWS가 관리 |
| 장애 대응 | 직접 이중화 설계 필요 | 리전, AZ 기반 설계 가능 |
| 운영 속도 | 장비 준비 시간이 필요 | 빠르게 생성, 변경, 삭제 가능 |

여기서 중요한 것은 클라우드가 항상 무조건 좋다는 뜻이 아니라는 점이다. 클라우드는 잘 쓰면 유연하고 빠르지만, 설정을 잘못하거나 리소스를 방치하면 비용과 보안 문제가 발생할 수 있다.

### 3.3 클라우드 도입 시 기대 효과

| 기대 효과 | 의미 | 이해 포인트 |
|---|---|---|
| 비용 효율성 | 필요한 만큼 사용하고 비용 지불 | 리소스 방치 시 비용 증가 가능 |
| 확장성 | 트래픽 증가에 맞춰 자원 증가 | 서버를 빠르게 추가할 수 있음 |
| 탄력성 | 사용량 감소 시 자원 축소 | 줄이는 능력까지 포함해야 클라우드답다 |
| 민첩성 | 빠르게 인프라 구성 가능 | 개발과 실험 속도가 빨라짐 |
| 글로벌 접근성 | 여러 리전에 서비스 배포 가능 | 사용자와 가까운 리전을 선택 가능 |
| 고가용성 | 장애에도 서비스 지속 가능 | AZ, 백업, 복구 설계와 연결됨 |

클라우드의 장점은 “싸다”가 아니라 “필요한 만큼 빠르게 조절할 수 있다”에 가깝다. 비용 절감은 자동으로 따라오는 결과가 아니라, 정확한 리소스 관리가 있을 때 가능한 결과다.

### 3.4 AWS 주요 서비스 영역

강의자료에서는 AWS 주요 서비스를 컴퓨팅, 스토리지, 데이터베이스, 네트워킹, 보안, 모니터링, DevOps 영역으로 나누어 소개했다.

| 영역 | 대표 서비스 | 내가 이해한 역할 |
|---|---|---|
| 컴퓨팅 | EC2, Lambda, EKS, ECS | 애플리케이션을 실행하는 영역 |
| 스토리지 | S3, EBS | 데이터를 저장하는 영역 |
| 데이터베이스 | RDS 등 | 데이터를 구조적으로 관리하는 영역 |
| 네트워킹 | VPC | 클라우드 내부 네트워크를 구성하는 영역 |
| 보안 | IAM, WAF, GuardDuty | 권한과 보안 위협을 관리하는 영역 |
| 모니터링 | CloudWatch, CloudTrail | 상태와 기록을 확인하는 영역 |
| DevOps | CodeBuild, CodeDeploy | 빌드와 배포를 자동화하는 영역 |

처음에는 EC2는 서버, S3는 저장소, IAM은 권한, VPC는 네트워크, CloudWatch는 모니터링 정도로 연결해두면 된다.

### 3.5 AWS 글로벌 인프라

AWS 글로벌 인프라는 리전, 가용 영역, 엣지 로케이션 등으로 구성된다.

| 구성 요소 | 의미 | 이해 포인트 |
|---|---|---|
| 리전 | 지리적으로 분리된 AWS 서비스 운영 지역 | 서울 리전은 `ap-northeast-2` |
| 가용 영역 | 하나의 리전 안에 있는 독립적인 데이터센터 그룹 | 장애 격리와 고가용성 설계에 중요 |
| 엣지 로케이션 | 사용자 가까이에서 콘텐츠를 제공하는 캐싱 인프라 | 지연 시간 감소와 CDN에 연결 |
| Local Zones | 사용자와 워크로드에 더 가까운 AWS 인프라 | 낮은 지연 시간이 필요한 서비스에 활용 |
| Outposts | 온프레미스에서 AWS 인프라와 서비스를 실행 | 하이브리드 클라우드와 연결 |
| Wavelength Zones | 5G 네트워크 엣지에 AWS 서비스를 배치 | 초저지연 모바일 서비스와 연결 |

리전 선택은 단순히 가까운 곳을 고르는 문제가 아니다. 사용자 위치, 서비스 지원 여부, 비용, 법적 규제까지 같이 고려해야 한다.

## 4. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| 온디맨드 | 클라우드 사용 방식의 핵심 | 필요할 때 생성하고 필요 없으면 제거 |
| 탄력성 | 클라우드 비용 관리와 연결 | 늘리는 것뿐 아니라 줄이는 것도 포함 |
| 리전 | 서비스 위치와 비용에 영향 | 리소스는 리전 단위로 생성·조회됨 |
| 가용 영역 | 장애 격리와 고가용성의 핵심 | 여러 AZ에 분산해야 장애에 강함 |
| 엣지 로케이션 | 사용자 응답 속도와 연결 | 콘텐츠를 사용자 가까이에서 제공 |
| 관리형 서비스 | 운영 부담을 줄임 | AWS가 일부 운영을 대신 담당 |
| 리전별 서비스 차이 | 실습과 설계에서 중요 | 모든 리전에서 모든 서비스가 되는 것은 아님 |

## 5. 실습하면서 내 것으로 만들 부분

1주차는 개념 중심 강의자료이며, 별도의 실습 명령어는 자료에 직접 제시되지 않았다. 따라서 이번 주차에서는 콘솔을 누르는 실습보다 앞으로 AWS 서비스를 볼 때 사용할 사고 흐름을 잡는 것이 중요하다.

```text
클라우드 개념 확인
→ 온프레미스와 비교
→ AWS 서비스 영역 확인
→ 리전/AZ/엣지 구조 이해
→ 리전 선택 기준 정리
→ 이후 실습에서 리소스 생성 위치와 비용 확인
```

### 5.1 파이프라인

1. 클라우드 컴퓨팅을 “인터넷 기반 온디맨드 IT 리소스 제공”으로 이해한다.
2. 온프레미스와 클라우드를 비용, 확장성, 유지보수 관점에서 비교한다.
3. AWS 서비스를 컴퓨팅, 스토리지, 네트워킹, 보안, 모니터링 영역으로 나눈다.
4. 리전과 AZ가 왜 필요한지 장애 대응 관점에서 본다.
5. 서비스를 배포할 때 리전 선택 기준을 확인한다.

### 5.2 명령어 또는 코드 흐름

이번 주차 강의자료에는 직접 실행할 CLI 명령어가 제시되어 있지 않다.

```bash
# 확실하지 않음: 1주차 자료에는 직접 실행 명령어가 제시되어 있지 않음
```

다만 이후 실습에서는 항상 다음 순서를 확인해야 한다.

```text
리전 확인
→ 서비스 선택
→ 리소스 생성
→ 상태 확인
→ 비용 발생 여부 확인
→ 필요 없으면 삭제
```

### 5.3 개념별로 정리

| 개념 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| 클라우드 컴퓨팅 | IT 자원을 직접 구매하지 않고 사용할 때 | 인프라를 서비스처럼 빌려 쓰는 방식 |
| 온프레미스 | 자체 서버를 직접 운영할 때 | 장비와 운영 책임을 직접 갖는 방식 |
| 리전 | 리소스를 어느 지역에 만들지 정할 때 | 서비스의 물리적 운영 지역 |
| AZ | 장애에 대비해 분산 배치할 때 | 한 리전 안의 독립된 인프라 단위 |
| 엣지 로케이션 | 사용자 응답 속도를 줄일 때 | 사용자 가까이에서 콘텐츠를 제공하는 지점 |

### 5.4 헷갈렸던 지점

- 클라우드는 무조건 비용이 싼 것이 아니다.
- 리전과 AZ는 같은 개념이 아니다.
- AWS가 모든 보안을 대신 책임지는 것은 아니다.
- 서비스가 글로벌해 보여도 리소스는 특정 리전에 생성되는 경우가 많다.
- 관리형 서비스도 사용자의 설정 책임이 남아 있다.

| 개념 A | 개념 B | 차이점 |
|---|---|---|
| 리전 | 가용 영역 | 리전은 지역, AZ는 리전 안의 독립 인프라 단위 |
| 확장성 | 탄력성 | 확장성은 늘리는 능력, 탄력성은 늘리고 줄이는 능력 |
| 비용 절감 | 비용 최적화 | 비용 절감은 줄이는 것, 최적화는 필요한 만큼 쓰는 것 |
| 온프레미스 | 클라우드 | 직접 소유와 서비스형 사용의 차이 |

### 5.5 나중에 다시 볼 포인트

- 클라우드 컴퓨팅을 내 말로 설명할 수 있는가?
- 온프레미스와 클라우드의 차이를 비용, 확장성, 운영 책임 관점에서 비교할 수 있는가?
- EC2, S3, IAM, VPC, CloudWatch가 각각 어떤 문제를 해결하는지 구분할 수 있는가?
- 리전 선택 기준을 사용자 위치, 비용, 법적 규제, 서비스 지원 여부로 나눠 설명할 수 있는가?

## 6. 보충 설명

* 자료 기반 내용:
  * 클라우드는 IT 리소스를 인터넷을 통해 온디맨드로 제공하고, 사용한 만큼 비용을 지불하는 방식이다.
  * AWS 주요 서비스는 컴퓨팅, 스토리지, 데이터베이스, 네트워킹, 보안, 모니터링, DevOps 영역으로 나뉜다.
  * AWS 글로벌 인프라는 리전, AZ, 엣지 로케이션 등을 통해 안정성과 성능을 제공한다.

* 보충 설명:
  * 클라우드 비용은 리소스를 잘 삭제하고 모니터링할 때만 효율적으로 관리된다.
  * 리전 선택은 기술 문제이면서 동시에 규제, 비용, 사용자 경험 문제다.

* 확실하지 않음:
  * 1주차 자료만으로는 실제 AWS 계정 생성 절차나 CLI 설정 절차는 확인되지 않는다.

## 7. 한 줄 요약

클라우드 컴퓨팅은 IT 인프라를 직접 소유하는 방식에서 벗어나, 필요한 자원을 온디맨드로 사용하고 글로벌 인프라 위에서 유연하게 운영하는 방식이다.
