# 클라우드활용 6주차: S3와 VPC 기초 이해 및 실습

## 1. 이번 주차에서 다룬 내용

이번 주차에서는 AWS의 대표적인 스토리지 서비스인 S3와 네트워크 기본 단위인 VPC를 학습했다. S3는 객체 스토리지이고, VPC는 AWS 리소스를 배치하는 가상 네트워크다. 둘 다 클라우드 운영에서 매우 자주 쓰이는 기본 서비스다.

이번 주차에서 다룬 내용은 다음과 같다.

- Amazon S3의 개념과 주요 특징
- 버킷, 객체, 스토리지 클래스
- S3 버킷 단위 권한과 보안 설정
- VPC의 개념과 구성 요소
- 퍼블릭 서브넷과 프라이빗 서브넷
- 라우트 테이블과 인터넷 게이트웨이
- 네트워크 ACL과 보안 그룹 비교
- S3 버킷 생성, 업로드, 퍼미션 설정 흐름
- VPC와 서브넷 생성 흐름

## 2. Background

- 객체 스토리지:
  파일을 객체 단위로 저장하는 스토리지 방식이다. 각 객체는 데이터와 메타데이터를 함께 가진다.

- 버킷:
  S3에서 객체를 담는 최상위 컨테이너다.

- 객체:
  S3에 저장되는 실제 파일 단위다.

- VPC:
  AWS 안에서 사용자가 정의하는 가상 네트워크다.

- 서브넷:
  VPC 안에서 IP 범위를 더 작게 나눈 네트워크 영역이다.

- 라우트 테이블:
  네트워크 트래픽이 어디로 갈지 결정하는 규칙이다.

### 보충 설명

S3와 VPC는 성격이 완전히 다르다. S3는 데이터를 저장하는 서비스이고, VPC는 네트워크 경계를 만드는 서비스다. 하지만 실제 클라우드 아키텍처에서는 둘을 따로 생각하면 안 된다. 예를 들어 EC2는 VPC 안에 생성되고, 애플리케이션은 S3에 파일을 저장할 수 있다. 즉, 컴퓨팅, 스토리지, 네트워크는 항상 함께 묶인다.

## 3. 개념 정리

### 3.1 Amazon S3

Amazon S3는 객체 스토리지 서비스다. 파일을 버킷에 객체 형태로 저장하며, 강의자료에서는 S3가 3개 이상의 가용 영역에 객체를 중복 저장해 높은 내구성을 제공한다고 정리한다.

| 개념 | 의미 | 이해 포인트 |
|---|---|---|
| 버킷 | 객체를 담는 컨테이너 | 리전 단위로 생성 |
| 객체 | S3에 저장되는 파일 단위 | 데이터와 메타데이터 포함 |
| 키 | 객체의 경로 또는 이름 | 폴더처럼 보이는 구조를 만들 수 있음 |
| 스토리지 클래스 | 저장 비용과 접근 빈도에 따른 저장 방식 | 자주 접근하는지에 따라 선택 |
| 내구성 | 데이터가 손실되지 않는 정도 | 여러 AZ 중복 저장과 연결 |

S3를 사용할 때 중요한 것은 “저장소를 만들었다”가 아니라 “누가 이 객체에 접근할 수 있는가”까지 같이 확인하는 것이다.

### 3.2 S3 권한과 보안 설정

강의자료에서는 S3 권한을 버킷 정책과 IAM 정책 관점으로 다룬다.

| 권한 방식 | 의미 | 이해 포인트 |
|---|---|---|
| 버킷 정책 | S3 버킷 전체에 적용되는 JSON 기반 정책 | 리소스 중심 정책 |
| IAM 정책 | 사용자나 역할에 적용되는 권한 정책 | 사용자 중심 정책 |
| 퍼블릭 접근 차단 | 외부 공개를 막는 보호 장치 | 실수로 공개되는 사고 방지 |

S3는 편리하지만 공개 설정을 잘못하면 데이터 유출로 이어질 수 있다. 따라서 버킷을 생성한 뒤에는 퍼블릭 접근 설정과 정책을 반드시 확인해야 한다.

### 3.3 VPC

VPC는 AWS 리소스를 담는 가상 네트워크다. EC2 같은 리소스는 VPC 안에 배치된다. VPC를 만들 때는 리전과 CIDR 주소 범위를 선택한다.

| 구성 요소 | 의미 | 이해 포인트 |
|---|---|---|
| VPC | AWS 안의 가상 네트워크 | 리소스가 위치하는 네트워크 경계 |
| CIDR | IP 주소 범위 표기법 | VPC와 서브넷 주소 범위 결정 |
| 서브넷 | VPC 내부의 세부 네트워크 | 퍼블릭/프라이빗으로 나뉠 수 있음 |
| 인터넷 게이트웨이 | VPC와 인터넷 연결 | 퍼블릭 서브넷 구성에 필요 |
| 라우트 테이블 | 트래픽 경로 제어 | `0.0.0.0/0 → IGW`가 핵심 |
| 보안 그룹 | 인스턴스 단위 방화벽 | 상태 저장 방식 |
| NACL | 서브넷 단위 방화벽 | 허용과 거부 규칙 가능 |

### 3.4 퍼블릭 서브넷과 프라이빗 서브넷

| 구분 | 퍼블릭 서브넷 | 프라이빗 서브넷 |
|---|---|---|
| 인터넷 접근 | 인터넷 게이트웨이로 직접 가능 | 직접 인터넷 접근 불가 |
| 라우트 테이블 | `0.0.0.0/0 → IGW` 경로 존재 | 외부 직접 경로 없음 또는 NAT 사용 |
| 배치 대상 | 웹 서버, 로드밸런서 | DB, 내부 서버 |
| 보안 관점 | 외부 접근 통제가 중요 | 내부 보호에 적합 |

퍼블릭 서브넷은 인터넷과 직접 연결될 수 있기 때문에 보안 그룹과 NACL 설정이 특히 중요하다. 프라이빗 서브넷은 외부에 직접 노출하지 않아야 하는 리소스를 배치할 때 사용한다.

### 3.5 NACL과 보안 그룹 비교

| 항목 | 보안 그룹 | NACL |
|---|---|---|
| 적용 단위 | 인스턴스 수준 | 서브넷 수준 |
| 상태 | Stateful | Stateless |
| 규칙 | 허용 규칙 중심 | 허용/거부 규칙 가능 |
| 평가 방식 | 모든 규칙을 종합 | 번호 순서대로 평가 |
| 사용 목적 | 인스턴스 접근 제어 | 서브넷 경계 제어 |

보안 그룹과 NACL은 둘 다 방화벽처럼 보이지만 적용 위치와 동작 방식이 다르다. 보안 그룹은 EC2 인스턴스 앞에서 동작하고, NACL은 서브넷 경계에서 동작한다고 이해하면 된다.

## 4. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| 객체 스토리지 | S3의 저장 방식 이해 | 파일을 객체 단위로 저장 |
| 버킷 정책 | S3 공개 사고 방지 | 리소스 중심 권한 설정 |
| CIDR | VPC 주소 설계에 필요 | IP 범위를 표현하는 방식 |
| IGW | 퍼블릭 서브넷 구성 핵심 | 인터넷으로 나가는 경로 |
| 라우트 테이블 | 네트워크 흐름 결정 | 목적지와 대상의 조합 |
| NACL | 서브넷 단위 제어 | Stateless 동작 이해 필요 |
| Security Group | 인스턴스 단위 제어 | Stateful 동작 이해 필요 |

## 5. 실습하면서 내 것으로 만들 부분

이번 주차는 S3와 VPC를 각각 만들어보는 흐름이다. 중요한 것은 단순히 버킷과 VPC를 생성하는 것이 아니라, 접근 권한과 네트워크 경로를 함께 확인하는 것이다.

```text
S3 버킷 생성
→ 객체 업로드
→ 버킷 권한 확인
→ 퍼블릭 접근 차단 확인
→ VPC 생성
→ CIDR 설정
→ 서브넷 생성
→ 인터넷 게이트웨이 연결
→ 라우트 테이블 설정
→ 보안 그룹/NACL 차이 확인
```

### 5.1 파이프라인

1. S3 버킷을 생성한다.
2. 파일을 객체로 업로드한다.
3. 버킷 정책과 퍼블릭 접근 차단 설정을 확인한다.
4. VPC를 생성하고 CIDR 범위를 지정한다.
5. 퍼블릭 서브넷과 프라이빗 서브넷을 나눈다.
6. 인터넷 게이트웨이를 연결한다.
7. 퍼블릭 서브넷의 라우트 테이블에 `0.0.0.0/0 → IGW`를 추가한다.
8. 보안 그룹과 NACL이 어디에 적용되는지 비교한다.

### 5.2 명령어 또는 코드 흐름

강의자료는 콘솔 기반 데모 중심으로 구성되어 있다. 라우트 테이블의 핵심 구조는 다음처럼 이해할 수 있다.

```text
Destination   Target
10.0.0.0/16   local
0.0.0.0/0     IGW
```

위 구조에서 `10.0.0.0/16 → local`은 VPC 내부 통신이고, `0.0.0.0/0 → IGW`는 외부 인터넷으로 나가는 기본 경로다.

### 5.3 개념별로 정리

| 개념 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| S3 Bucket | 객체를 저장할 공간을 만들 때 | 파일을 담는 최상위 컨테이너 |
| Object | 파일을 S3에 저장할 때 | S3에 저장되는 실제 데이터 단위 |
| Bucket Policy | 버킷 접근 권한을 제어할 때 | 버킷 중심의 JSON 정책 |
| VPC | 클라우드 네트워크를 만들 때 | AWS 리소스를 담는 가상 네트워크 |
| Subnet | 네트워크를 나눌 때 | 퍼블릭/프라이빗 영역 구분 |
| IGW | 인터넷 연결이 필요할 때 | VPC와 인터넷 연결 통로 |
| Route Table | 트래픽 경로를 정할 때 | 목적지별 다음 경로 설정 |
| NACL | 서브넷 단위 방화벽이 필요할 때 | 서브넷 경계 접근 제어 |
| Security Group | 인스턴스 보호가 필요할 때 | 인스턴스 단위 접근 제어 |

### 5.4 헷갈렸던 지점

- S3는 폴더 구조처럼 보여도 실제로는 객체 키 기반이다.
- 버킷 정책과 IAM 정책은 모두 JSON이지만 적용 대상이 다르다.
- 퍼블릭 서브넷은 이름 때문이 아니라 라우트 테이블에 IGW 경로가 있어야 퍼블릭해진다.
- 보안 그룹과 NACL은 둘 다 방화벽 같지만 적용 위치와 상태 관리 방식이 다르다.
- VPC를 만들 때 CIDR 범위를 대충 정하면 나중에 네트워크 확장 시 문제가 될 수 있다.

| 개념 A | 개념 B | 차이점 |
|---|---|---|
| 버킷 정책 | IAM 정책 | 버킷 정책은 리소스 중심, IAM 정책은 사용자/역할 중심 |
| 퍼블릭 서브넷 | 프라이빗 서브넷 | IGW로 향하는 기본 경로가 있느냐가 핵심 |
| 보안 그룹 | NACL | 보안 그룹은 인스턴스 단위, NACL은 서브넷 단위 |
| Stateful | Stateless | Stateful은 응답 트래픽 자동 허용, Stateless는 양방향 규칙 필요 |

### 5.5 나중에 다시 볼 포인트

- S3 버킷과 객체의 차이를 설명할 수 있는가?
- 버킷 정책과 IAM 정책의 적용 대상 차이를 설명할 수 있는가?
- 퍼블릭 서브넷을 만들기 위해 필요한 요소를 말할 수 있는가?
- 보안 그룹과 NACL의 차이를 상태 관리 방식까지 포함해서 설명할 수 있는가?

## 6. 보충 설명

* 자료 기반 내용:
  * S3는 3개 이상의 가용 영역에 객체를 중복 저장하여 높은 내구성을 제공한다.
  * S3 버킷은 리전에 생성하고, 버킷에 파일을 객체로 업로드한다.
  * VPC 생성 시 리전과 CIDR 표기법으로 private 주소 범위를 결정한다.
  * IGW로 향하는 라우트 테이블 설정으로 퍼블릭 서브넷을 구성할 수 있다.
  * NACL은 서브넷 수준, 보안 그룹은 인스턴스 수준의 가상 방화벽이다.

* 보충 설명:
  * S3 공개 설정은 보안 사고로 이어질 수 있으므로 실습 후 반드시 접근 권한을 확인해야 한다.
  * 퍼블릭/프라이빗 서브넷 구분은 이름이 아니라 라우트 테이블 구조로 판단해야 한다.

* 확실하지 않음:
  * 6주차 자료만으로는 실제 실습에서 사용한 정확한 버킷 이름, VPC CIDR, 서브넷 CIDR 값은 확인되지 않는다.

## 7. 한 줄 요약

S3는 객체를 저장하는 서비스이고, VPC는 AWS 리소스가 배치되는 가상 네트워크이며, 둘 다 권한과 경로 설정을 함께 이해해야 안전하게 사용할 수 있다.
