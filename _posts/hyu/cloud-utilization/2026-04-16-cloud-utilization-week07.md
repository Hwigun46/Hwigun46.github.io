# 클라우드활용 7주차: AWS 보안 책임 공유 모델과 WAF, GuardDuty

## 1. 이번 주차에서 다룬 내용

이번 주차에서는 AWS 보안의 기본 구조와 보안 서비스를 학습했다. 4주차에서 IAM을 통해 권한 관리를 배웠고, 6주차에서 VPC와 보안 그룹을 배웠다면, 7주차는 이 보안 요소들이 클라우드 책임 공유 모델 안에서 어떤 의미를 갖는지 정리하는 주차다.

이번 주차에서 다룬 내용은 다음과 같다.

- 클라우드 보안 책임 공유 모델
- 사용자가 직접 관리해야 하는 보안 항목
- IAM, VPC, 보안 그룹 등 기본 보안 구성 요소
- 보안 서비스 도입의 필요성
- AWS WAF의 역할과 적용 대상
- 웹 공격 유형과 WAF 구성 요소
- Amazon GuardDuty의 목적과 위협 탐지 방식
- CloudWatch Events, Lambda, SNS를 활용한 대응 흐름

## 2. Background

- 책임 공유 모델:
  AWS와 고객이 보안 책임을 나누어 갖는 모델이다.

- WAF:
  웹 애플리케이션 앞단에서 웹 공격을 필터링하는 보안 서비스다.

- GuardDuty:
  AWS 환경의 이상 행위와 위협 징후를 자동으로 탐지하는 서비스다.

- CloudTrail:
  AWS API 호출 기록을 남기는 서비스다.

- AWS Config:
  리소스 설정 변경을 추적하고 규정 준수 상태를 확인하는 서비스다.

### 보충 설명

클라우드 보안에서 가장 위험한 오해는 “AWS를 쓰면 보안은 AWS가 알아서 해준다”는 생각이다. AWS는 물리적 인프라와 클라우드 자체 보안을 담당하지만, 사용자가 만든 리소스의 권한, 데이터 공개 설정, 네트워크 접근, 운영체제 패치 같은 부분은 고객 책임이다.

## 3. 개념 정리

### 3.1 클라우드 보안 책임 공유 모델

AWS에서는 보안 책임을 AWS와 고객이 나누어 가진다. 강의자료에서는 물리적 서버 보안은 AWS 책임이고, S3 버킷 접근 권한 설정, 인스턴스 OS 보안 패치, 데이터 암호화 키 관리는 고객 책임으로 구분한다.

| 항목 | AWS 책임 | 고객 책임 | 이해 포인트 |
|---|---|---|---|
| 물리적 서버 보안 | O | X | 데이터센터와 하드웨어는 AWS가 관리 |
| S3 버킷 접근 권한 설정 | X | O | 공개 설정 실수는 고객 책임 |
| 인스턴스 OS 보안 패치 | X | O | EC2 내부 OS 관리는 고객 책임 |
| 데이터 암호화 키 관리 | 일부 제공 | O | KMS 설정과 키 운영은 고객 책임 |

또한 강의자료에서는 다음 사례를 고객 책임으로 본다.

| 상황 | 책임 주체 |
|---|---|
| EC2 포트 22가 열려 있음 | 고객 |
| RDS 비밀번호 노출 | 고객 |
| VPC 라우팅 설정 오류 | 고객 |
| AWS 내부 하드웨어 결함 | AWS |

결국 사용자는 “내가 설정한 것”에 대한 책임을 피할 수 없다.

### 3.2 기본 보안 구성 요소

| 구성 요소 | 역할 | 이해 포인트 |
|---|---|---|
| IAM | 사용자, 역할, 정책 기반 접근 제어 | 누가 무엇을 할 수 있는지 결정 |
| VPC | 네트워크 경계 구성 | 리소스를 격리하고 라우팅 제어 |
| 보안 그룹 | 인스턴스 단위 가상 방화벽 | 허용 규칙 기반, Stateful |
| CloudTrail | API 호출 기록 | 누가 어떤 작업을 했는지 추적 |
| AWS Config | 리소스 설정 변경 추적 | 보안 설정 위반 탐지 |

보안은 하나의 서비스만 켠다고 완성되지 않는다. IAM으로 권한을 줄이고, VPC로 네트워크를 나누고, 보안 그룹으로 포트를 제한하고, CloudTrail과 Config로 변경을 추적해야 한다.

### 3.3 보안 서비스 도입의 필요성

강의자료에서는 보안 서비스들을 다음과 같이 정리한다.

| 서비스 | 기능 | 목적 | 특징 |
|---|---|---|---|
| AWS WAF | 웹 공격 필터링 | 웹 보안 | 규칙 기반 제어 |
| GuardDuty | 지능형 위협 탐지 | 이상 행동 탐지 | 머신러닝 기반 |
| Shield | DDoS 방어 | 서비스 가용성 보장 | 기본형 무료 제공 |
| CloudTrail | API 호출 기록 | 행동 감사 및 추적 | 보안 사고 분석 |
| AWS Config | 리소스 설정 변경 추적 | 규정 준수 및 감시 | 지속적인 감사 |

이 서비스들은 각각 다른 보안 문제를 담당한다. WAF는 웹 요청을 막고, GuardDuty는 이상 행위를 탐지하며, CloudTrail은 행위 기록을 남긴다.

### 3.4 AWS WAF

WAF는 웹 애플리케이션 앞단에서 공격 요청을 필터링하는 서비스다. XSS, SQL Injection 같은 웹 공격을 규칙 기반으로 차단할 수 있다.

| WAF 구성 요소 | 의미 | 이해 포인트 |
|---|---|---|
| Rule | 요청을 검사하는 조건 | 특정 패턴을 차단하거나 허용 |
| Rule Group | 여러 Rule의 묶음 | 재사용 가능한 규칙 모음 |
| Web ACL | 실제 보호 대상에 연결하는 정책 | ALB, CloudFront 등에 적용 |

WAF 적용 대상은 다음과 같이 정리할 수 있다.

| 적용 대상 | 설명 |
|---|---|
| Amazon CloudFront | 글로벌 CDN 앞단에서 웹 공격 방어 |
| Application Load Balancer | EC2 기반 웹 애플리케이션 트래픽 제어 |
| Amazon API Gateway | API 호출에 대한 필터링 |

### 3.5 Amazon GuardDuty

GuardDuty는 AWS 환경에서 이상 징후를 자동으로 탐지하고 보안 알림을 제공하는 서비스다. 수동으로 모든 로그를 사람이 보는 것은 현실적으로 어렵기 때문에, GuardDuty 같은 탐지 서비스가 필요하다.

GuardDuty 경고 이후 대응 흐름은 다음처럼 이해할 수 있다.

| 도구 | 설명 |
|---|---|
| Amazon CloudWatch Events | 탐지 이벤트를 실시간 전달 |
| AWS Lambda | 자동 대응 스크립트 실행 |
| SNS | 관리자에게 알림 메일 또는 문자 전송 |

즉, GuardDuty는 탐지 자체를 담당하고, CloudWatch Events/EventBridge, Lambda, SNS는 탐지 이후 알림과 대응 자동화에 연결된다.

| GuardDuty | 이상 행위 탐지 | 로그 기반 자동 위협 탐지 |

## 5. 실습하면서 내 것으로 만들 부분

이번 주차는 보안 서비스를 켜는 것보다, 어느 계층에서 어떤 보안 문제가 발생하고 어떤 서비스가 대응하는지 연결하는 것이 중요하다.

```text
책임 공유 모델 확인
→ 고객 책임 영역 구분
→ IAM/VPC/보안 그룹 점검
→ WAF 적용 대상 확인
→ GuardDuty 활성화 흐름 이해
→ 탐지 이벤트 확인
→ 알림 또는 자동 대응 구조 연결
```

### 5.1 파이프라인

1. 현재 보안 문제가 AWS 책임인지 고객 책임인지 구분한다.
2. IAM에서 최소 권한이 적용되어 있는지 확인한다.
3. VPC와 보안 그룹에서 불필요한 포트가 열려 있는지 확인한다.
4. 웹 서비스라면 WAF 적용 대상을 확인한다.
5. Web ACL과 Rule을 구성한다.
6. GuardDuty를 활성화하고 탐지 결과를 확인한다.
7. 탐지 이벤트를 CloudWatch Events/EventBridge, Lambda, SNS와 연결하는 흐름을 이해한다.

### 5.2 명령어 또는 코드 흐름

강의자료는 콘솔 기반 개념과 서비스 흐름 중심이며, 직접 실행 CLI 명령어는 확인되지 않는다.

```bash
# 확실하지 않음: 7주차 자료에는 WAF 또는 GuardDuty CLI 명령어가 직접 제시되어 있지 않음
```

보안 점검 흐름은 다음처럼 정리할 수 있다.

```text
EC2 보안 그룹 확인
→ 22/3389 포트가 전체 공개인지 확인
→ S3 버킷 퍼블릭 접근 확인
→ CloudTrail 기록 확인
→ GuardDuty Findings 확인
→ 필요 시 SNS/Lambda로 알림 또는 차단 자동화
```

### 5.3 개념별로 정리

| 개념 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| Shared Responsibility Model | 보안 책임 범위를 나눌 때 | AWS 책임과 고객 책임 구분 |
| Security Group | EC2 접근을 제한할 때 | 인스턴스 단위 가상 방화벽 |
| AWS WAF | 웹 공격을 차단할 때 | 웹 요청 앞단의 필터 |
| Web ACL | WAF 규칙을 적용할 때 | Rule을 묶어 보호 대상에 연결 |
| GuardDuty | 이상 행위를 탐지할 때 | 자동 위협 탐지 서비스 |
| CloudTrail | 누가 어떤 API를 호출했는지 볼 때 | 감사 로그 |
| AWS Config | 설정 변경과 규정 준수를 볼 때 | 지속적인 설정 감시 |
| SNS | 보안 알림을 보낼 때 | 관리자 알림 채널 |
| Lambda | 자동 대응을 구현할 때 | 이벤트 기반 대응 실행 |

### 5.4 헷갈렸던 지점

- AWS 보안 책임 공유 모델에서 “클라우드의 보안”과 “클라우드 안의 보안”은 다르다.
- WAF는 네트워크 방화벽이 아니라 웹 요청 레벨의 필터다.
- GuardDuty는 공격을 직접 차단하는 서비스라기보다 탐지와 경고에 가깝다.
- CloudTrail은 보안 서비스라기보다 감사와 추적에 필요한 로그 기록이다.
- 보안 그룹이 열려 있으면 WAF가 있어도 SSH 노출 같은 문제는 따로 남는다.

| 개념 A | 개념 B | 차이점 |
|---|---|---|
| WAF | Security Group | WAF는 웹 요청 필터, Security Group은 인스턴스 네트워크 접근 제어 |
| GuardDuty | CloudTrail | GuardDuty는 탐지, CloudTrail은 API 기록 |
| Shield | WAF | Shield는 DDoS 방어, WAF는 웹 공격 필터링 |
| AWS 책임 | 고객 책임 | 물리 인프라는 AWS, 설정과 데이터 보호는 고객 |

### 5.5 나중에 다시 볼 포인트

- S3 버킷 공개 설정이 왜 고객 책임인지 설명할 수 있는가?
- EC2 22번 포트를 전체 공개하면 왜 위험한가?
- WAF가 막을 수 있는 공격과 막기 어려운 영역을 구분할 수 있는가?
- GuardDuty 탐지 이후 알림과 자동 대응 흐름을 설명할 수 있는가?

## 6. 보충 설명

* 자료 기반 내용:
  * AWS에서는 물리적 인프라는 AWS가 책임지고, 데이터와 접근 제어는 사용자가 책임진다.
  * 보안 그룹은 EC2 등 인스턴스에 적용되는 가상 방화벽이며 Stateful 방식으로 동작한다.
  * WAF는 웹 애플리케이션 앞단에서 다양한 공격을 실시간으로 차단하는 보안 필터다.
  * GuardDuty는 AWS 환경에서 자동으로 이상 징후를 탐지하고 보안 알림을 제공한다.

* 보충 설명:
  * 보안 서비스는 단독으로 쓰기보다 로그, 알림, 자동 대응과 함께 연결해야 실무적인 가치가 생긴다.
  * GuardDuty 탐지 결과는 사람이 확인하는 것에서 끝나면 대응 속도가 늦기 때문에 EventBridge/Lambda/SNS와 연결하는 구성이 중요하다.

* 확실하지 않음:
  * 7주차 자료만으로는 실제 WAF Rule 세부 조건이나 GuardDuty Finding 예시 값 전체는 확인되지 않는다.

## 7. 한 줄 요약

AWS 보안은 책임 공유 모델을 기준으로 고객이 관리해야 할 영역을 구분하고, IAM·VPC·보안 그룹·WAF·GuardDuty를 조합해 예방과 탐지를 함께 구성하는 것이다.
