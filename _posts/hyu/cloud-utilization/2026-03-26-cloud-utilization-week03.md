# 클라우드활용 3주차: AWS 비용 구조와 과금 모델

## 1. 이번 주차에서 다룬 내용

이번 주차에서는 AWS 비용 구조와 과금 모델을 학습했다. 클라우드는 사용한 만큼 비용을 지불하는 구조이기 때문에, 리소스를 만드는 방법만큼 비용이 어떻게 발생하는지 이해하는 것이 중요하다.

이번 주차에서 다룬 내용은 다음과 같다.

- AWS 사용량 기반 과금 구조
- 온디맨드, Savings Plans/예약 인스턴스, 스팟 인스턴스 비교
- 서비스별 과금 방식 사례
- AWS 프리 티어의 개념과 유의사항
- Billing Dashboard, Cost Explorer, Budget을 활용한 비용 관리

## 2. Background

- 사용량 기반 과금:
  리소스를 사용한 시간, 용량, 요청 수, 데이터 전송량 등에 따라 비용이 발생하는 방식이다.

- 온디맨드:
  약정 없이 필요한 만큼 사용하는 요금 모델이다.

- 예약 인스턴스/Savings Plans:
  일정 기간 사용을 약정하고 할인받는 요금 모델이다.

- 스팟 인스턴스:
  AWS의 남는 자원을 저렴하게 사용하는 방식이지만 중단될 수 있다.

- 프리 티어:
  신규 사용자나 특정 서비스에 대해 일정 한도 내에서 무료 사용량을 제공하는 제도다.

### 보충 설명

클라우드 비용은 “서버를 켜는 순간”부터만 발생한다고 단순하게 생각하면 부족하다. EC2 실행 시간뿐 아니라 EBS 볼륨, S3 저장 용량, 데이터 전송량, WAF 규칙, 로그 저장 등 다양한 요소에서 비용이 발생할 수 있다.

따라서 클라우드 실습에서는 리소스 생성보다 더 중요한 것이 비용 확인과 리소스 정리다.

## 3. 개념 정리

### 3.1 클라우드 과금 구조

AWS는 사용량 기반 종량제 과금 모델을 기본으로 한다. 사용자는 필요한 서비스를 사용하고, 실제 사용량에 따라 비용을 지불한다.

이 방식은 초기 비용을 줄이는 장점이 있지만, 사용량을 추적하지 않으면 예상하지 못한 비용이 발생할 수 있다. 특히 실습 환경에서는 EC2 인스턴스, EBS 볼륨, 퍼블릭 IP, 데이터 전송, 로그 저장 등이 비용으로 이어질 수 있다.

### 3.2 요금 모델 비교

| 요금 모델 | 특징 | 적합한 경우 | 주의할 점 |
|---|---|---|---|
| 온디맨드 | 사용 시간 단위로 비용 지불, 계약 없음 | 예측하기 어려운 워크로드 | 단가는 상대적으로 높음 |
| Savings Plans/예약 인스턴스 | 1년 또는 3년 장기 약정으로 할인 | 장기적이고 안정적인 워크로드 | 사용량 예측이 틀리면 낭비 가능 |
| 스팟 인스턴스 | 남는 인스턴스 자원을 저렴하게 사용 | 중단 가능하거나 유연한 작업 | 회수될 수 있고 2분 전 공지 가능 |

내가 이해한 방식으로 정리하면, 온디맨드는 “편하지만 비싼 선택”, 예약/Savings Plans는 “예측 가능할 때 저렴한 선택”, 스팟은 “끊겨도 되는 작업에 쓰는 선택”이다.

### 3.3 프리 티어

AWS 프리 티어는 클라우드 입문자를 위한 무료 체험 요금제다. 주요 서비스를 일정 한도 내에서 사용할 수 있지만, 무료 한도를 초과하면 비용이 발생한다.

프리 티어를 사용할 때는 다음을 계속 확인해야 한다.

- 무료 제공 기간
- 월별 무료 사용량
- 무료 대상 인스턴스 타입
- EBS, S3 등 추가 리소스 사용량
- 데이터 전송량
- 리전별 조건 차이

프리 티어는 “아무거나 무료”가 아니라 “정해진 조건 안에서만 무료”다.

### 3.4 비용 분석 도구

| 도구 | 역할 | 내가 이해한 의미 |
|---|---|---|
| Billing Dashboard | 현재 사용량과 청구 정보를 확인 | 비용 현황판 |
| Cost Explorer | 서비스별, 기간별 비용 분석 | 비용 그래프 분석 도구 |
| Budget | 예산 설정과 알림 구성 | 비용 초과 방지 장치 |
| 태그 기반 비용 분석 | 프로젝트/조직 단위 비용 구분 | 누가 얼마나 썼는지 나누는 기준 |

강의자료에서는 현재 월 사용량, 서비스별 비용 분포, 태그별 비용, 지출 예측, 결제 이력 등을 비용 관리 항목으로 다룬다.

### 3.5 서비스별 과금 방식

AWS 서비스는 모두 같은 방식으로 과금되지 않는다.

| 서비스 | 주요 과금 기준 | 이해 포인트 |
|---|---|---|
| EC2 | 인스턴스 실행 시간, 인스턴스 타입, 스토리지 | 켜둔 시간과 사양이 중요 |
| S3 | 저장 용량, 요청 수, 데이터 전송 | 파일을 저장만 해도 비용 가능 |
| WAF | Web ACL, 룰, 요청 수 | 보안 설정도 비용 요소가 될 수 있음 |
| CloudWatch | 로그 저장, 지표, 알람 | 모니터링 데이터도 비용 관리 필요 |

## 4. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| 종량제 과금 | 클라우드 비용 구조의 기본 | 사용하지 않는 리소스도 조건에 따라 비용 발생 |
| 프리 티어 한도 | 실습 비용 방지에 중요 | 무료 조건을 넘으면 과금됨 |
| 스팟 인스턴스 | 비용 절감에 유리 | 중단되어도 되는 작업에 적합 |
| Budget | 비용 초과 방지 | 알림을 먼저 걸어두는 습관 필요 |
| Cost Explorer | 비용 원인 분석 | 어느 서비스가 비용을 만들었는지 확인 |
| 태그 | 비용 분리와 관리 | 프로젝트별 비용 분석에 필요 |

## 5. 실습하면서 내 것으로 만들 부분

이번 주차에서 중요한 실습 흐름은 리소스를 생성하는 것이 아니라, 비용이 어디서 발생하는지 확인하고 예산 알림을 설정하는 것이다.

```text
요금 모델 확인
→ 프리 티어 조건 확인
→ Billing Dashboard 확인
→ Cost Explorer에서 서비스별 비용 분석
→ Budget 생성
→ 알림 기준 설정
→ 실습 후 리소스 정리
```

### 5.1 파이프라인

1. AWS Pricing 페이지에서 서비스별 요금 구조를 확인한다.
2. 프리 티어 조건을 확인한다.
3. Billing Dashboard에서 현재 월 사용량을 확인한다.
4. Cost Explorer에서 서비스별 비용 분포를 본다.
5. Budget을 생성하고 월 예산을 설정한다.
6. 예산 초과 예상 시 알림을 받도록 구성한다.
7. 실습 후 EC2, EBS, S3 등 남아 있는 리소스를 확인한다.

### 5.2 명령어 또는 코드 흐름

강의자료에는 비용 관리 도구 사용이 콘솔 중심으로 제시되어 있으며, 구체적인 CLI 명령어는 확인되지 않는다.

```bash
# 확실하지 않음: 3주차 자료에는 비용 관리 CLI 명령어가 직접 제시되어 있지 않음
```

실습 흐름은 다음처럼 정리할 수 있다.

```text
Billing and Cost Management 접속
→ Billing Dashboard 확인
→ Cost Explorer 열기
→ 기간/서비스별 비용 확인
→ Budgets 메뉴에서 예산 생성
→ 알림 기준과 이메일 설정
```

### 5.3 개념별로 정리

| 개념 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| 온디맨드 | 예측 어려운 작업을 할 때 | 약정 없이 쓰는 대신 비쌀 수 있음 |
| 예약/Savings Plans | 장기 사용량이 안정적일 때 | 약정으로 비용을 낮추는 방식 |
| 스팟 | 중단되어도 되는 배치 작업에 쓸 때 | 저렴하지만 회수될 수 있음 |
| 프리 티어 | 학습용 실습을 할 때 | 무료 한도 안에서만 안전함 |
| Billing Dashboard | 전체 비용을 볼 때 | 현재 비용 상태 확인 |
| Cost Explorer | 비용 원인을 분석할 때 | 서비스별 비용 흐름 확인 |
| Budget | 비용 상한을 관리할 때 | 예산 초과 전 알림 받기 |

### 5.4 헷갈렸던 지점

- 프리 티어는 계정 전체가 무제한 무료라는 뜻이 아니다.
- EC2를 중지해도 EBS 비용은 남을 수 있다.
- 스팟 인스턴스는 저렴하지만 안정적인 운영 서버에는 적합하지 않을 수 있다.
- 비용은 인스턴스 실행 시간만이 아니라 스토리지, 요청 수, 데이터 전송에서도 발생한다.
- Budget은 비용을 자동으로 막아주는 기능이 아니라 알림을 주는 관리 도구로 이해해야 한다.

| 개념 A | 개념 B | 차이점 |
|---|---|---|
| 온디맨드 | 예약 인스턴스 | 온디맨드는 유연성, 예약은 장기 할인 |
| 예약 인스턴스 | 스팟 인스턴스 | 예약은 안정적 사용, 스팟은 중단 가능 저비용 |
| Billing Dashboard | Cost Explorer | 대시보드는 현황 확인, Cost Explorer는 분석 |
| Budget | 비용 차단 | Budget은 알림 중심이며 자동 차단과는 다름 |

### 5.5 나중에 다시 볼 포인트

- 온디맨드, 예약, 스팟을 각각 어떤 상황에 써야 하는가?
- 프리 티어가 끝나거나 한도를 넘으면 어떤 비용이 발생할 수 있는가?
- Billing Dashboard와 Cost Explorer에서 무엇을 확인해야 하는가?
- 예산 알림을 설정하는 이유를 설명할 수 있는가?

## 6. 보충 설명

* 자료 기반 내용:
  * AWS는 사용량 기반 종량제 과금 모델을 기본으로 한다.
  * 온디맨드, Savings Plans/예약 인스턴스, 스팟 인스턴스는 특징과 적합한 사용 상황이 다르다.
  * Billing Dashboard와 Cost Explorer를 통해 비용을 시각적으로 분석할 수 있다.
  * Budget을 설정하면 예산 초과를 사전에 감지할 수 있다.

* 보충 설명:
  * 실습용 계정에서는 리소스 삭제 습관이 가장 중요하다.
  * 비용 최적화는 단순히 싼 요금제를 고르는 것이 아니라 워크로드 특성에 맞는 요금 모델을 선택하는 일이다.

* 확실하지 않음:
  * 3주차 자료만으로는 실제 Budget 생성 화면의 세부 입력값까지는 확인되지 않는다.

## 7. 한 줄 요약

AWS 비용 관리는 리소스를 만들기 전에 요금 모델을 이해하고, 만든 뒤에는 Billing Dashboard, Cost Explorer, Budget으로 지속적으로 확인하는 과정이다.
