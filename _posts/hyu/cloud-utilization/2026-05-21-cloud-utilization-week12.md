# 클라우드활용 12주차: 멀티 AZ와 ELB 고가용성 설계

이번 주차의 핵심은 서비스가 장애 상황에서도 계속 동작하도록 구성하는 고가용성 설계이다. 클라우드에서는 하나의 서버가 정상적으로 동작하는 것보다, 일부 리소스가 실패해도 전체 서비스가 유지되는 구조가 더 중요하다.

## 고가용성

고가용성은 서비스를 가능한 한 오랫동안 중단 없이 제공하는 능력이다.

장애가 아예 발생하지 않는 시스템을 만드는 것은 현실적으로 어렵다. 그래서 고가용성 설계는 장애가 발생해도 빠르게 복구하거나 자동으로 다른 리소스로 우회하여 서비스 지속성을 유지하는 것을 목표로 한다.

## 가용성

가용성은 서비스가 정상적으로 사용 가능한 시간의 비율이다.

| 가용성 | 1년 기준 중단 시간 | 의미 |
|---|---|---|
| 99% | 약 87시간 | 중단이 자주 발생할 수 있음 |
| 99.9% | 약 8시간 45분 | 일반적인 서비스 수준 |
| 99.99% | 약 52분 | 높은 가용성 수준 |
| 99.999% | 약 5분 | 매우 민감한 시스템 수준 |

가용성이 높아질수록 설계 난이도와 비용도 함께 증가한다.

## 리전

리전은 AWS가 전 세계에 운영하는 지리적 위치 단위이다.

예를 들어 서울 리전은 `ap-northeast-2`로 표현된다. 리전은 데이터 위치, 지연 시간, 법적 요구사항, 장애 격리 등을 고려해 선택한다.

## 가용 영역

가용 영역은 하나의 리전 안에 존재하는 독립적인 데이터센터 묶음이다.

각 가용 영역은 독립된 전원, 냉각, 보안 시스템을 갖고 있으며, 다른 가용 영역과 빠른 네트워크로 연결된다. 하나의 AZ에 장애가 발생해도 다른 AZ는 영향을 덜 받도록 설계된다.

## 멀티 AZ

멀티 AZ는 두 개 이상의 가용 영역에 애플리케이션 구성 요소를 분산하거나 복제하는 방식이다.

하나의 AZ에만 서버와 데이터베이스를 배치하면 해당 AZ 장애 시 서비스가 중단될 수 있다. 반면 멀티 AZ 구조를 사용하면 한 AZ에 문제가 생겨도 다른 AZ의 리소스를 통해 서비스를 계속 제공할 수 있다.

## 멀티 AZ 적용 예시

| 서비스 | 멀티 AZ 방식 |
|---|---|
| RDS | 주 인스턴스와 대기 인스턴스를 다른 AZ에 배치 |
| EC2 | 여러 AZ에 인스턴스를 분산 배포 |
| EFS | 여러 AZ에서 접근 가능한 파일 스토리지 제공 |
| ELB | 여러 AZ의 정상 인스턴스로 트래픽 분산 |

## ELB

ELB는 Elastic Load Balancing의 약자로, 들어오는 트래픽을 여러 대상에게 분산하는 AWS 로드밸런서 서비스이다.

서버 한 대에 모든 요청이 몰리면 과부하가 발생할 수 있고, 서버 하나가 장애를 일으키면 전체 서비스가 중단될 수 있다. ELB는 여러 인스턴스에 트래픽을 나누고, 정상 상태의 인스턴스에만 요청을 전달한다.

## 로드밸런서

로드밸런서는 사용자 요청을 여러 서버로 나누어 보내는 장치 또는 서비스이다.

로드밸런서를 사용하면 특정 서버에 트래픽이 집중되는 것을 막고, 장애가 발생한 서버를 트래픽 대상에서 제외할 수 있다. 고가용성 구조에서 로드밸런서는 거의 필수적인 구성요소이다.

## Target Group

Target Group은 로드밸런서가 트래픽을 전달할 대상들의 묶음이다.

EC2 인스턴스, IP 주소, Lambda 함수 등이 대상이 될 수 있다. Auto Scaling Group과 연결하면 새로 생성된 인스턴스가 자동으로 Target Group에 등록될 수 있다.

## ELB 유형

| 유형 | 특징 | 사용 사례 |
|---|---|---|
| ALB | L7 기반, 경로·호스트 기반 라우팅 | 웹 애플리케이션, REST API |
| NLB | L4 기반, 빠른 처리와 고정 IP 지원 | 실시간 게임, 금융 서비스 |
| CLB | 이전 세대 로드밸런서 | 레거시 시스템 |

일반적인 웹 서비스에서는 ALB를 가장 많이 사용한다.

## 헬스 체크

헬스 체크는 ELB가 대상 인스턴스의 상태를 주기적으로 확인하는 기능이다.

ELB는 지정된 경로에 요청을 보내고 응답 상태를 확인한다. 응답이 실패하면 해당 인스턴스를 Unhealthy 상태로 보고 트래픽 대상에서 제외한다. 이후 정상 응답이 다시 확인되면 Healthy 상태로 복귀시킨다.

## 장애 복구 흐름

멀티 AZ와 ELB를 함께 사용하면 장애 상황에서 자동 우회가 가능하다.

사용자는 ELB로 접속하고, ELB는 여러 AZ의 인스턴스 상태를 확인한다. 특정 인스턴스나 AZ에 문제가 생기면 정상 인스턴스로 트래픽을 보내 서비스 중단 가능성을 줄인다.

## 한 줄 정리

멀티 AZ와 ELB는 하나의 서버나 하나의 가용 영역에 장애가 발생해도 서비스가 계속 유지되도록 만드는 고가용성 설계의 핵심이다.
