# 클라우드활용 13주차: AWS 클라우드 보안 실무 사례

이번 주차의 핵심은 클라우드 환경에서 발생할 수 있는 보안 위협을 이해하고, 사고를 예방하기 위한 기본 대응 전략을 정리하는 것이다. 클라우드 보안은 특정 보안 장비 하나로 해결되는 문제가 아니라, 계정·권한·설정·로그·정책을 함께 관리하는 문제이다.

## 클라우드 보안 위협

클라우드 환경에서는 리소스와 API가 인터넷을 통해 접근 가능하므로 공격 표면이 넓어진다.

물리적으로 서버에 접근하지 않아도 잘못된 설정, 유출된 자격 증명, 과도한 권한, 공개된 스토리지, 취약한 API를 통해 보안 사고가 발생할 수 있다.

## 온프레미스와 클라우드 보안 차이

온프레미스 환경에서는 물리적 보안과 망 분리가 중요한 비중을 차지한다.

클라우드 환경에서는 논리적 격리, 권한 관리, 설정 관리, API 접근 통제가 더 중요하다. 특히 클라우드는 설정 하나로 리소스가 외부에 노출될 수 있기 때문에 기본 설정을 그대로 믿으면 위험하다.

## 구성 오류

구성 오류는 클라우드 보안 사고의 대표적인 원인이다.

예를 들어 S3 버킷을 공개로 설정하거나, IAM 권한을 과도하게 부여하거나, 보안 그룹을 너무 넓게 열어두면 의도하지 않은 접근이 가능해질 수 있다. 클라우드에서는 리소스 생성이 쉽기 때문에 잘못된 설정도 빠르게 확산될 수 있다.

## 자격 증명 유출

자격 증명 유출은 Access Key, 비밀번호, 토큰 같은 인증 정보가 외부로 노출되는 상황이다.

Access Key가 코드 저장소에 업로드되거나 약한 비밀번호를 사용하면 공격자가 정상 사용자처럼 AWS 리소스에 접근할 수 있다. 자격 증명이 유출되면 API 호출, 리소스 생성, 데이터 삭제 같은 피해가 발생할 수 있다.

## 계정 도용

계정 도용은 공격자가 정상 사용자의 계정을 탈취하여 클라우드 자원을 악용하는 공격이다.

피싱, Credential Stuffing, 약한 비밀번호, 유출된 Access Key가 원인이 될 수 있다. 계정이 도용되면 공격자는 리소스를 생성하거나 데이터를 삭제하거나 비용을 발생시킬 수 있다.

## 데이터 유출

데이터 유출은 S3, 데이터베이스, 로그, 백업 파일 등에 저장된 민감 정보가 외부로 노출되는 상황이다.

클라우드에서는 스토리지 접근 정책, 암호화 설정, IAM 권한, 네트워크 접근 제어가 모두 데이터 보호와 연결된다.

## 악의적인 내부자

악의적인 내부자는 조직 내부 권한을 가진 사용자가 의도적으로 데이터를 유출하거나 리소스를 악용하는 경우이다.

내부자는 이미 권한을 가지고 있기 때문에 외부 공격자보다 탐지가 어려울 수 있다. 최소 권한 원칙, 로그 추적, 권한 분리, 정기적인 접근 검토가 필요하다.

## 서비스 거부 공격

DoS 또는 DDoS는 서비스가 정상 요청을 처리하지 못하도록 과도한 트래픽을 발생시키는 공격이다.

클라우드에서는 Auto Scaling, ELB, WAF, Shield 같은 서비스를 활용해 대응할 수 있지만, 비용 증가와 서비스 품질 저하까지 함께 고려해야 한다.

## 안전하지 않은 인터페이스와 API

클라우드는 대부분 API를 통해 제어된다.

API 인증이 약하거나, 권한 검증이 부족하거나, 관리 콘솔 접근이 제대로 제한되지 않으면 공격자가 리소스를 조작할 수 있다. 따라서 API 접근 제어와 로그 추적이 중요하다.

## 기본 보안 대응 전략

클라우드 보안의 기본은 예방, 탐지, 대응을 함께 구성하는 것이다.

| 전략 | 의미 |
|---|---|
| MFA 적용 | 계정 로그인 보호 |
| 최소 권한 원칙 | 필요한 권한만 부여 |
| Access Key 관리 | 키 노출 방지와 주기적 교체 |
| 로그 기록 | CloudTrail로 사용자 행위 추적 |
| 설정 점검 | S3 공개 설정, IAM 권한, 보안 그룹 확인 |
| 자동 탐지 | GuardDuty, Config, EventBridge 활용 |
| 암호화 | 저장 데이터와 전송 데이터 보호 |

## S3 공개 사고

S3 버킷을 잘못 공개하면 민감 데이터가 외부에 노출될 수 있다.

S3는 편리한 객체 스토리지이지만, 접근 정책과 퍼블릭 액세스 차단 설정을 제대로 관리하지 않으면 데이터 유출로 이어질 수 있다.

## 루트 계정 탈취

루트 계정은 AWS 계정의 최상위 권한을 가진다.

루트 계정이 탈취되면 모든 리소스에 접근할 수 있기 때문에 MFA를 필수로 설정하고, 평소 운영 작업에는 루트 계정을 사용하지 않는 것이 안전하다.

## 과도한 IAM 권한

IAM 권한을 넓게 부여하면 사용자가 실수하거나 계정이 탈취되었을 때 피해 범위가 커진다.

`AdministratorAccess`처럼 강한 권한은 꼭 필요한 경우에만 사용해야 하며, 서비스와 작업 단위로 권한을 좁히는 것이 중요하다.

## AWS 기본 보안 서비스

| 서비스 | 역할 |
|---|---|
| IAM | 사용자, 그룹, 역할, 권한 관리 |
| CloudTrail | API 호출과 사용자 행위 기록 |
| GuardDuty | 위협 탐지 |
| WAF | 웹 공격 방어 |
| Config | 리소스 설정 변경과 규정 준수 점검 |
| EventBridge | 이벤트 기반 자동 대응 |

## 한 줄 정리

클라우드 보안 사고는 대부분 잘못된 설정, 권한 관리 부주의, 자격 증명 유출에서 시작되므로 최소 권한·로그 추적·설정 점검·자동 탐지가 기본이다.
