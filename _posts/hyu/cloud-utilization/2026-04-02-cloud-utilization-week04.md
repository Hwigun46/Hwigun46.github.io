# 클라우드활용 4주차: IAM을 활용한 사용자 권한 관리와 보안 기초

## 1. 이번 주차에서 다룬 내용

이번 주차에서는 AWS 보안의 핵심인 IAM을 학습했다. IAM은 AWS에서 “누가 어떤 리소스에 어떤 작업을 할 수 있는가”를 제어하는 서비스다. 클라우드에서는 리소스를 빠르게 만들 수 있는 만큼, 권한을 잘못 주면 비용 문제와 보안 사고가 바로 발생할 수 있다.

이번 주차에서 다룬 내용은 다음과 같다.

- 접근 제어의 필요성
- IAM 사용자, 그룹, 역할, 정책 개념
- IAM 사용자와 그룹 생성 흐름
- IAM 정책의 JSON 구조
- Allow, Deny, Implicit Deny의 차이
- 최소 권한 원칙
- ARN의 의미
- IAM Role과 임시 권한

## 2. Background

- 인증:
  사용자가 누구인지 확인하는 과정이다.

- 인가:
  인증된 사용자가 어떤 작업을 할 수 있는지 결정하는 과정이다.

- 접근 제어:
  특정 사용자나 서비스가 리소스에 접근할 수 있는 범위를 제한하는 것이다.

- JSON:
  AWS IAM 정책을 표현하는 텍스트 기반 데이터 형식이다.

- ARN:
  AWS 리소스를 고유하게 식별하는 문자열이다.

### 보충 설명

IAM은 단순히 “사용자 만드는 기능”이 아니다. IAM의 핵심은 권한의 경계를 정확히 만드는 것이다. 사용자는 누구인지, 그룹은 어떤 권한 묶음인지, 역할은 누가 임시로 맡을 수 있는 권한인지, 정책은 어떤 행동을 허용하거나 거부하는 규칙인지 구분해야 한다.

## 3. 개념 정리

### 3.1 IAM 구성 요소

| 구성 요소 | 설명 | 이해 포인트 |
|---|---|---|
| 사용자(User) | AWS 리소스에 접근할 수 있는 개별 계정 | 사람 또는 시스템 단위 신원 |
| 그룹(Group) | 여러 사용자를 묶어 공통 권한을 부여 | 권한 관리를 단순화 |
| 역할(Role) | 사용자나 서비스가 임시로 맡을 수 있는 신원 | 장기 키 대신 임시 권한 사용 |
| 정책(Policy) | JSON 형식으로 정의된 권한 집합 | 허용/거부 규칙의 실제 내용 |

IAM을 공부할 때 사용자와 역할을 혼동하기 쉽다. 사용자는 고정된 신원이고, 역할은 특정 상황에서 임시로 맡는 권한이다.

### 3.2 IAM 정책의 평가 방식

IAM 정책에서는 명시적 거부가 가장 강하다.

| 평가 방식 | 의미 | 예시 |
|---|---|---|
| Explicit Deny | 명시적으로 거부하면 어떤 Allow보다 우선 | `ec2:TerminateInstances` 금지 |
| Explicit Allow | 명시적으로 허용된 작업만 가능 | `s3:GetObject` 허용 |
| Implicit Deny | 아무 권한도 없으면 기본적으로 거부 | 정책에 없는 작업은 불가 |

내가 이해한 방식으로 정리하면, IAM은 기본적으로 “안 된다”에서 출발한다. 필요한 권한만 Allow로 열어주고, 반드시 막아야 하는 위험 작업은 Deny로 닫는다.

### 3.3 최소 권한 원칙

최소 권한 원칙은 사용자나 서비스에게 필요한 권한만 부여하는 것이다.

예를 들어 개발자에게 EC2 시작/중지 권한만 필요하다면 `ec2:StartInstances`, `ec2:StopInstances` 정도만 주어야 한다. EC2 삭제 권한이나 IAM 관리 권한까지 부여하면 실수나 계정 탈취 시 피해 범위가 커진다.

권한은 편하게 주면 당장은 작업이 빠르지만, 장기적으로는 보안 위험이 커진다.

### 3.4 IAM 정책 JSON 구조

IAM 정책은 JSON 형식으로 작성된다.

| 항목 | 설명 |
|---|---|
| `Version` | 정책 언어의 버전, 일반적으로 `2012-10-17` 사용 |
| `Effect` | `Allow` 또는 `Deny` |
| `Action` | 허용하거나 거부할 작업 |
| `Resource` | 정책이 적용될 AWS 리소스 |
| `Condition` | 선택 항목, 특정 IP나 MFA 등 조건 기반 제한 |

정책은 결국 `Effect + Action + Resource` 조합으로 이해하면 된다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

위 예시는 특정 S3 버킷 객체를 읽을 수 있도록 허용하는 형태다.

### 3.5 ARN

ARN은 AWS 리소스를 고유하게 식별하기 위한 문자열이다.

```text
arn:partition:service:region:account-id:resource
```

| 요소 | 의미 | 예시 |
|---|---|---|
| `arn` | 고정 문자열 | 항상 `arn`으로 시작 |
| `partition` | AWS 파티션 | `aws`, `aws-cn`, `aws-us-gov` |
| `service` | 서비스 이름 | `s3`, `ec2`, `iam` |
| `region` | 리전 | `ap-northeast-2` |
| `account-id` | AWS 계정 ID | `123456789012` |
| `resource` | 리소스 이름 또는 경로 | 버킷명, 인스턴스 ID 등 |

ARN은 정책에서 “어떤 리소스에 대한 권한인지”를 지정할 때 필요하다.

### 3.6 IAM Role과 임시 권한

IAM Role은 사용자나 AWS 서비스가 임시로 수임할 수 있는 권한이다. 장기 Access Key를 직접 저장하는 방식보다 Role을 통해 임시 권한을 사용하는 방식이 보안상 더 낫다.

예를 들어 EC2가 S3에 접근해야 한다면, EC2 안에 Access Key를 하드코딩하는 것이 아니라 EC2에 IAM Role을 연결하는 방식이 더 안전하다.

## 4. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| Implicit Deny | IAM 기본 동작 이해에 필요 | 허용하지 않으면 기본적으로 거부 |
| Explicit Deny | 보안상 반드시 막아야 할 작업 제어 | 어떤 Allow보다 우선 |
| 최소 권한 원칙 | 권한 남용 방지 | 필요한 권한만 부여 |
| ARN | 정책의 대상 리소스 지정 | AWS 리소스의 고유 식별자 |
| IAM Role | 임시 권한 관리 | 장기 키 저장을 줄일 수 있음 |
| Condition | 세밀한 접근 제어 | IP, MFA 등 조건 기반 제한 가능 |

## 5. 실습하면서 내 것으로 만들 부분

이번 주차의 핵심은 사용자를 만들고 권한을 붙이는 절차를 외우는 것이 아니라, 권한이 어떤 흐름으로 평가되는지 이해하는 것이다.

```text
IAM 사용자/그룹 확인
→ 그룹 생성
→ 사용자 생성
→ 사용자를 그룹에 연결
→ 정책 생성 또는 기존 정책 선택
→ 그룹에 정책 연결
→ 새 사용자로 로그인
→ 권한 적용 확인
```

### 5.1 파이프라인

1. IAM 서비스에 접속한다.
2. IAM 사용자를 생성한다.
3. IAM 그룹을 생성한다.
4. 사용자를 그룹에 추가한다.
5. 그룹에 정책을 연결한다.
6. 생성한 사용자로 로그인한다.
7. 허용된 작업과 거부된 작업을 확인한다.

### 5.2 명령어 또는 코드 흐름

강의자료의 실습은 콘솔 중심이다. 정책 자체는 JSON으로 구성되므로, 정책 구조를 읽는 연습이 필요하다.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
```

이 정책은 특정 S3 버킷 목록 조회와 객체 읽기를 허용하는 구조로 이해할 수 있다.

### 5.3 코드/함수별로 정리

| 코드/항목 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| `Version` | IAM 정책 문법 버전을 지정할 때 | 정책 언어 버전 |
| `Effect` | 허용/거부 여부를 정할 때 | Allow 또는 Deny |
| `Action` | 어떤 작업을 허용/거부할지 정할 때 | 서비스 API 작업 |
| `Resource` | 어떤 리소스에 적용할지 정할 때 | ARN으로 대상 지정 |
| `Condition` | 특정 조건에서만 허용할 때 | IP, MFA 등 조건 제한 |
| `arn:aws:s3:::bucket/*` | S3 객체를 지정할 때 | 버킷 내부 객체 전체 |

### 5.4 헷갈렸던 지점

- 사용자와 역할은 같은 개념이 아니다.
- 그룹은 권한을 직접 실행하는 주체가 아니라 사용자에게 권한을 묶어 주는 단위다.
- 정책에 없으면 기본적으로 거부된다.
- Allow가 있어도 Explicit Deny가 있으면 거부된다.
- `Resource`에 버킷 자체와 버킷 내부 객체는 다르게 표현될 수 있다.

| 개념 A | 개념 B | 차이점 |
|---|---|---|
| User | Role | User는 고정 신원, Role은 임시로 맡는 권한 |
| Group | Policy | Group은 사용자 묶음, Policy는 권한 규칙 |
| Explicit Deny | Implicit Deny | Explicit Deny는 명시적 거부, Implicit Deny는 기본 거부 |
| Action | Resource | Action은 무엇을 할지, Resource는 어디에 할지 |

### 5.5 나중에 다시 볼 포인트

- IAM 사용자, 그룹, 역할, 정책을 구분할 수 있는가?
- IAM 정책에서 `Effect`, `Action`, `Resource`를 읽을 수 있는가?
- Explicit Deny가 왜 가장 우선인지 설명할 수 있는가?
- 최소 권한 원칙을 실제 권한 설계에 적용할 수 있는가?

## 6. 보충 설명

* 자료 기반 내용:
  * IAM은 사용자별로 세밀한 접근 제어를 가능하게 하는 AWS 보안 관리의 핵심이다.
  * IAM 정책은 JSON 형식으로 구성된다.
  * IAM 정책은 `Effect`, `Action`, `Resource`, `Condition` 등을 통해 권한을 정의한다.
  * 임시 권한은 IAM Role을 활용하는 것이 권장된다.

* 보충 설명:
  * 실무에서는 사용자를 직접 많이 만들기보다 역할, 그룹, SSO 등을 함께 고려한다.
  * Access Key를 코드에 직접 넣는 방식은 피해야 한다.

* 확실하지 않음:
  * 4주차 자료만으로는 실제 MFA 설정 절차나 AWS SSO 구성 절차는 확인되지 않는다.

## 7. 한 줄 요약
