# 클라우드프로그래밍 2주차: AWS 컴퓨팅 서비스와 EC2 기본 구조

## 1. 이번 주차에서 다룬 내용

2주차는 클라우드에서 서버를 어떻게 빌려 쓰는지 배우는 주차였다. 서버와 클라이언트의 요청/응답 구조에서 시작해서, 서버 OS, 가상화, EC2 인스턴스, 보안 그룹, 키 페어, 모니터링까지 이어진다.

이번 주차에서 다룬 내용은 다음과 같다.

- 서버와 클라이언트의 요청/응답 구조
- 웹 서버, DB 서버, 메일 서버의 역할
- 서버 OS와 리눅스/윈도우 서버
- 서버 가상화 개념
- Amazon EC2 인스턴스
- AMI, 인스턴스 유형, 스토리지, 네트워크, 보안 그룹, 키 페어
- EC2 배포와 SSH 접속 흐름

## 2. 왜 이 내용을 배우는가

클라우드에서 가장 기본이 되는 자원은 컴퓨팅 자원이다. AWS에서는 이 컴퓨팅 자원을 EC2 인스턴스 형태로 제공한다.

EC2를 제대로 이해하려면 서버가 무엇인지부터 알아야 한다. 서버는 요청을 받아 처리하고 응답을 돌려주는 역할을 한다. 그리고 그 서버가 클라우드에서는 물리 장비가 아니라 가상화된 인스턴스 형태로 제공된다.

이 주차를 대충 넘기면 뒤에서 VPC, Docker, Kubernetes를 배울 때 연결이 끊긴다. Docker 컨테이너도 결국 어떤 서버 위에서 실행되고, Kubernetes 노드도 결국 서버다. 따라서 EC2는 클라우드프로그래밍에서 가장 기본적인 실행 장소라고 볼 수 있다.

## 3. 먼저 알아야 할 배경지식

- 클라이언트: 서비스를 요청하는 프로그램
- 서버: 요청을 받아 처리하고 응답하는 컴퓨터 또는 프로그램
- HTTP/HTTPS: 웹 브라우저와 웹 서버가 데이터를 주고받을 때 사용하는 프로토콜
- OS: 하드웨어와 애플리케이션 사이에서 자원을 관리하는 소프트웨어
- 가상화: 하나의 물리 서버에서 여러 개의 가상 서버를 실행하는 기술
- SSH: 원격 리눅스 서버에 접속하기 위한 프로토콜

### 보충 설명

EC2는 단순히 AWS에서 만든 컴퓨터가 아니다. 물리 서버 위에 가상화 기술로 만들어진 서버 자원을 사용자가 인스턴스 단위로 빌려 쓰는 서비스다.

## 4. 강의자료 기반 개념 정리

### 4.1 서버와 클라이언트

서버는 네트워크에서 데이터나 서비스를 제공하는 컴퓨터이고, 클라이언트는 그 서비스를 이용하는 프로그램이다.

웹 브라우저가 웹 서버에 페이지를 요청하면 웹 서버는 HTML, CSS, 이미지 같은 데이터를 응답한다. 게임, SNS, 메일 서비스도 프로토콜은 다를 수 있지만 기본 구조는 요청과 응답이다.

처음에는 서버를 “큰 컴퓨터”라고 생각하기 쉽지만, 더 정확히는 역할이다. 같은 컴퓨터라도 어떤 상황에서는 서버가 되고, 어떤 상황에서는 클라이언트가 될 수 있다.

### 4.2 웹 서버, DB 서버, 메일 서버

서버는 역할에 따라 나눌 수 있다.

| 서버 종류 | 역할 | AWS에서 연결되는 서비스 |
|---|---|---|
| 웹 서버 | 웹 페이지와 웹 요청 처리 | EC2, ECS |
| DB 서버 | 데이터를 저장, 조회, 수정, 삭제 | RDS, DynamoDB |
| 메일 서버 | 메일 송신, 전달, 수신 | EC2, Amazon SES |

중요한 것은 서버 이름을 외우는 것이 아니라, 각 서버가 어떤 요청을 처리하는지 이해하는 것이다.

### 4.3 서버 OS

서버 OS는 서버에서 실행되는 운영체제다. 리눅스와 윈도우 서버가 대표적이다.

클라우드 서버도 결국 OS 위에서 동작한다. AWS 콘솔에서 버튼으로 인스턴스를 만들더라도, 실제 운영은 OS 위에서 패키지를 설치하고, 설정 파일을 수정하고, 로그를 확인하는 방식으로 진행된다.

### 4.4 가상화

가상화는 하나의 물리 서버 위에서 여러 개의 가상 서버를 실행하는 기술이다. EC2는 AWS의 대규모 서버에서 가상화를 수행하고, 사용자는 인스턴스라는 가상 서버를 생성해서 사용한다.

내가 이해한 방식으로 정리하면, 가상화는 큰 건물을 여러 개의 독립된 방처럼 나누어 쓰는 방식이다. 물리 자원은 공유하지만, 사용자는 자신만의 서버를 쓰는 것처럼 느낀다.

### 4.5 Amazon EC2

EC2는 AWS에서 가상 서버를 생성하고 사용하는 서비스다. 인스턴스를 만들 때는 다음 요소를 함께 결정해야 한다.

| 구성 요소 | 의미 |
|---|---|
| AMI | OS와 소프트웨어 구성이 담긴 템플릿 |
| 인스턴스 유형 | CPU, 메모리, 네트워크 성능 |
| 스토리지 | 데이터를 저장할 공간 |
| VPC | 인스턴스가 배치되는 네트워크 |
| 보안 그룹 | 인스턴스 송수신 트래픽 제어 |
| 키 페어 | SSH 접속을 위한 인증 키 |

강의자료에서는 프리 티어 실습에 `t2.micro` 인스턴스를 사용할 예정이라고 설명한다. 단, 프리 티어 조건은 계정 상태와 AWS 정책에 따라 달라질 수 있으므로 실제 실습 전에는 콘솔에서 다시 확인해야 한다.

## 5. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| AMI | EC2 생성의 출발점 | OS와 기본 소프트웨어가 담긴 템플릿 |
| 인스턴스 유형 | 성능과 비용 결정 | CPU, 메모리, 네트워크 성능 선택 |
| EBS | 서버 데이터 저장 | 인스턴스와 분리 가능한 블록 스토리지 |
| 보안 그룹 | 외부 접근 제어 | 인바운드/아웃바운드 규칙 |
| 키 페어 | SSH 접속 인증 | 비밀번호 대신 공개키/개인키 기반 접근 |
| CloudWatch | 운영 상태 확인 | 지표와 알람으로 모니터링 |
| Lambda | 서버리스 컴퓨팅 | 서버 관리 없이 코드 실행 |

## 6. 실습하면서 내 것으로 만들 부분

2주차 실습은 EC2 인스턴스를 만들고, SSH로 접속하고, 간단한 웹 서비스를 올려보는 흐름이다. 여기서 중요한 것은 “인스턴스를 만들었다”가 아니라, EC2 접속과 서비스 공개에 어떤 조건이 필요한지 체득하는 것이다.

### 6.1 실습 흐름을 내가 다시 정리하면

1. AMI를 선택한다.
2. 인스턴스 유형을 선택한다.
3. 네트워크와 보안 그룹을 설정한다.
4. 키 페어를 생성하거나 기존 키를 선택한다.
5. EC2 인스턴스를 시작한다.
6. SSH로 원격 서버에 접속한다.
7. 웹 서버를 설치하고 실행한다.
8. 브라우저나 `curl`로 웹 서버 응답을 확인한다.
9. 실습이 끝나면 인스턴스와 관련 자원을 정리한다.

이 흐름은 Git으로 비유하면 `git add` 후 `git commit`으로 기록을 남기고 `git push`로 원격에 올리는 흐름처럼, 단계마다 역할이 있다. EC2도 AMI, 인스턴스 유형, 키 페어, 보안 그룹, SSH 접속이 각각 따로 의미를 가진다.

### 6.2 EC2 접속 흐름에서 자주 쓰는 명령어

강의자료는 EC2 배포와 접근 흐름을 설명하지만, 세부 리눅스 명령어 전체가 명시되어 있지는 않다. 아래 명령어는 EC2 실습에서 일반적으로 같이 쓰는 보충 정리다.

```bash
# 개인키 파일 권한을 제한한다.
chmod 400 key.pem

# 개인키를 사용해 EC2에 SSH 접속한다.
ssh -i key.pem ubuntu@퍼블릭IP

# 패키지 목록을 갱신한다.
sudo apt update

# 웹 서버를 설치한다. Ubuntu 기준 예시다.
sudo apt install nginx -y

# 웹 서버 상태를 확인한다.
systemctl status nginx

# 내 서버가 HTTP 응답을 주는지 확인한다.
curl http://localhost
```

Amazon Linux 계열 AMI를 사용하면 `apt` 대신 `yum` 또는 `dnf`를 사용할 수 있다. 실제 명령어는 선택한 AMI에 따라 달라진다.

### 6.3 명령어별로 정리

| 명령어 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| `chmod 400 key.pem` | SSH 개인키 권한이 너무 열려 있을 때 | 개인키를 나만 읽을 수 있게 제한 |
| `ssh -i key.pem ubuntu@퍼블릭IP` | EC2 리눅스 서버에 접속할 때 | 내 터미널을 원격 서버 터미널로 연결 |
| `sudo apt update` | 패키지 설치 전 목록을 갱신할 때 | 설치 가능한 패키지 정보를 최신화 |
| `sudo apt install nginx -y` | 웹 서버를 설치할 때 | EC2를 웹 요청에 응답할 수 있는 서버로 준비 |
| `systemctl status nginx` | 웹 서버 실행 상태를 볼 때 | 프로세스가 정상 실행 중인지 확인 |
| `curl http://localhost` | 서버 내부에서 웹 응답을 확인할 때 | 웹 서버가 HTTP 응답을 주는지 확인 |

### 6.4 EC2에서 접속이 안 될 때 먼저 볼 것

EC2 실습에서 가장 많이 막히는 부분은 SSH 또는 웹 접속이다. 이때는 무작정 다시 만들기보다 조건을 나누어 봐야 한다.

| 문제 상황 | 먼저 확인할 것 |
|---|---|
| SSH 접속 실패 | 키 페어, 퍼블릭 IP, 보안 그룹 22번 포트 |
| 브라우저 접속 실패 | 보안 그룹 80번 포트, 웹 서버 실행 상태 |
| 서버 내부에서는 되는데 외부에서 안 됨 | 퍼블릭 IP, 퍼블릭 서브넷, 라우팅 |
| 인스턴스는 켜졌는데 비용이 걱정됨 | 실행 중 상태, EBS 볼륨, 탄력적 IP |

### 6.5 헷갈렸던 지점

EC2 인스턴스가 실행 중이라고 해서 외부에서 바로 접속할 수 있는 것은 아니다. 외부 접속에는 최소한 퍼블릭 IP, 올바른 라우팅, 보안 그룹 허용 규칙이 필요하다.

또한 인스턴스를 중지한다고 해서 모든 비용이 사라지는 것도 아니다. EBS 볼륨 같은 별도 자원은 남을 수 있다.

### 6.6 나중에 다시 볼 포인트

- EC2 접속은 키 페어와 보안 그룹이 모두 맞아야 한다.
- HTTP 접속은 80번 또는 443번 포트 허용이 필요하다.
- 퍼블릭 IP가 있어야 외부 인터넷에서 접근할 수 있다.
- 인스턴스 종료와 중지는 다르다.
- 실습 후에는 관련 리소스 삭제 여부를 반드시 확인해야 한다.
