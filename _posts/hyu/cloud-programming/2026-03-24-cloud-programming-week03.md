# 클라우드프로그래밍 3주차: 네트워크 기초와 AWS VPC 구조

## 1. 이번 주차에서 다룬 내용

3주차는 클라우드 서버가 어떻게 서로 통신하고, 외부 인터넷과 어떻게 연결되는지 배우는 주차였다. EC2를 만들었다고 해서 자동으로 안전하게 서비스가 공개되는 것은 아니다. IP, 포트, 라우팅, 보안 규칙이 맞아야 실제 통신이 된다.

이번 주차에서 다룬 내용은 다음과 같다.

- 네트워킹의 의미
- OSI 7계층
- IPv4와 IPv6
- 퍼블릭 IP와 프라이빗 IP
- 서브넷과 서브넷 마스크
- 라우팅과 라우터
- TCP, UDP, 포트 번호
- Packet Tracer, Wireshark, traceroute, whois 실습 흐름
- AWS VPC, 서브넷, 보안 그룹, 네트워크 ACL
- 인터넷 게이트웨이와 NAT 게이트웨이

## 2. Background

- IP 주소: 네트워크에서 대상을 식별하는 주소
- 포트: 한 서버 안에서 어떤 서비스로 연결할지 구분하는 번호
- 프로토콜: 통신 규칙
- 라우팅: 목적지까지 가는 경로 선택
- 방화벽: 허용하거나 차단할 트래픽을 결정하는 보안 장치
- CIDR: IP 범위를 표현하는 방식

### 보충 설명

네트워크는 단순히 인터넷 연결 여부만 의미하지 않는다. 주소 지정, 경로 선택, 포트 구분, 패킷 전달, 보안 제어가 모두 포함된다.

## 3. 개념 정리

### 3.1 네트워킹

네트워킹은 IT 자원들이 서로 연결되어 통신할 수 있는 환경을 의미한다.

웹 서버, DB 서버, 사용자 브라우저, 외부 API, 클라우드 서비스는 모두 네트워크를 통해 연결된다. 네트워크가 없으면 서버가 존재해도 서비스가 동작하지 않는다.

내가 이해한 방식으로는 네트워크는 클라우드 인프라의 도로망이다. 서버가 건물이라면 IP는 주소이고, 라우팅 테이블은 길 안내 표지판이며, 보안 그룹은 출입문 규칙이다.

### 3.2 OSI 7계층

OSI 7계층은 네트워크 통신 과정을 계층으로 나누어 설명하는 모델이다.

실무에서 모든 계층을 항상 자세히 외울 필요는 없지만, 문제가 발생했을 때 어느 계층의 문제인지 나누어 생각하는 데 도움이 된다.

| 계층 | 클라우드 실습에서 연결되는 부분 |
|---|---|
| 3계층 | IP 주소, 라우팅, VPC, 서브넷 |
| 4계층 | TCP/UDP, 포트 번호, 보안 그룹 |
| 7계층 | HTTP, DNS, SMTP 같은 애플리케이션 프로토콜 |

예를 들어 EC2 웹 서버 접속이 안 된다면 3계층에서는 라우팅과 IP를 보고, 4계층에서는 80번 포트 허용 여부를 보고, 7계층에서는 웹 서버가 실제로 응답하는지 봐야 한다.

### 3.3 IP 주소와 서브넷

IP 주소는 네트워크에서 대상을 식별하는 주소다. IPv4는 `192.168.0.1`처럼 네 개의 숫자로 표현된다.

서브넷은 큰 네트워크를 더 작은 네트워크 범위로 나눈 것이다. 클라우드에서는 VPC 안에 여러 서브넷을 만들고, 각 서브넷의 역할을 다르게 둔다.

- 퍼블릭 서브넷: 외부 인터넷과 직접 연결될 수 있는 영역
- 프라이빗 서브넷: 외부에서 직접 접근하지 못하게 두는 내부 영역

### 3.4 TCP, UDP, 포트

TCP와 UDP는 전송 계층에서 사용하는 대표적인 프로토콜이다.

- TCP: 연결을 맺고 신뢰성 있게 데이터를 전달한다.
- UDP: 연결 과정이 단순하고 빠르지만 신뢰성 보장은 약하다.

포트는 한 서버 안에서 어떤 서비스로 연결할지 구분하는 번호다.

| 포트 | 일반적인 용도 |
|---|---|
| 22 | SSH |
| 80 | HTTP |
| 443 | HTTPS |
| 3306 | MySQL |
| 5432 | PostgreSQL |

EC2에 보안 그룹을 설정할 때 “어떤 포트를 누구에게 열 것인가”를 결정해야 한다.

### 3.5 AWS VPC

VPC는 AWS 안에서 논리적으로 격리된 네트워크 공간이다. EC2 인스턴스는 VPC 안의 서브넷에 배치된다.

VPC를 구성할 때는 다음 요소가 함께 연결된다.

| 구성 요소 | 역할 |
|---|---|
| VPC | 전체 네트워크 범위 |
| 서브넷 | VPC 안의 세부 네트워크 구역 |
| 라우팅 테이블 | 목적지에 따른 경로 지정 |
| 인터넷 게이트웨이 | VPC와 인터넷 연결 |
| NAT 게이트웨이 | 프라이빗 서버의 외부 요청 통로 |
| 보안 그룹 | 인스턴스 단위 방화벽 |
| 네트워크 ACL | 서브넷 단위 방화벽 |

### 3.6 NAT 게이트웨이

NAT 게이트웨이는 프라이빗 서브넷의 서버가 외부 인터넷으로 나갈 수 있게 해주는 통로다. 중요한 점은 NAT 게이트웨이가 외부에서 프라이빗 서버로 들어오는 통로가 아니라는 점이다.

프라이빗 서버는 외부에서 직접 접근하지 못하게 두고, 필요한 업데이트나 외부 API 호출만 NAT를 통해 나가게 설계한다.

## 4. 내가 몰랐을 가능성이 높은 개념

| 개념 | 왜 중요한가 | 이해 포인트 |
|---|---|---|
| CIDR | VPC와 서브넷 주소 범위 설정 | `10.0.0.0/16`처럼 IP 범위 표현 |
| 라우팅 테이블 | 통신 경로 결정 | 목적지에 따라 다음 경로 지정 |
| 인터넷 게이트웨이 | 퍼블릭 통신에 필요 | VPC와 인터넷을 연결하는 관문 |
| NAT 게이트웨이 | 프라이빗 서버의 외부 요청 처리 | 외부 접근 허용이 아니라 내부에서 외부로 나가는 통로 |
| 보안 그룹 | 인스턴스 단위 방화벽 | 허용 규칙 중심의 상태 저장 방식 |
| 네트워크 ACL | 서브넷 단위 방화벽 | 허용/거부 규칙을 명시할 수 있음 |

## 5. 실습하면서 내 것으로 만들 부분

3주차 실습은 네트워크가 실제로 어떻게 보이는지 확인하는 과정이다. Packet Tracer, Wireshark, traceroute, whois 같은 도구는 각각 보는 위치가 다르다.

- Packet Tracer는 네트워크 구성을 시뮬레이션한다.
- Wireshark는 실제 패킷을 본다.
- traceroute는 목적지까지의 경로를 본다.
- whois는 IP나 도메인의 등록 정보를 본다.

### 5.1 파이프라인

1. Packet Tracer로 NAT 네트워크 구조를 구성한다.
2. 퍼블릭 IP와 프라이빗 IP의 차이를 확인한다.
3. 서브넷과 서브넷 마스크가 IP 범위를 어떻게 나누는지 본다.
4. Wireshark로 실제 패킷을 관찰한다.
5. `traceroute`로 목적지까지의 경로를 확인한다.
6. whois 검색으로 IP나 도메인 등록 정보를 확인한다.
7. AWS VPC에서 퍼블릭/프라이빗 서브넷 구조와 연결해 생각한다.

### 5.2 네트워크 확인 명령어 흐름

```bash
# 대상 서버까지 연결이 되는지 확인
ping example.com

# 목적지까지 어떤 경로를 거치는지 확인
traceroute example.com

# IP 또는 도메인의 등록 정보 확인
whois example.com
```

아래 명령어는 보충 설명이다. 실제 실습 환경에 따라 설치가 필요할 수 있다.

```bash
# 내 리눅스 서버의 IP 주소 확인
ip addr

# DNS 해석이 되는지 확인
nslookup example.com

# 외부에서 보이는 내 공인 IP 확인
curl ifconfig.me
```

### 5.3 명령어별로 정리

| 명령어 | 언제 쓰는가 | 내가 이해한 의미 |
|---|---|---|
| `ping 대상` | 네트워크 연결 자체가 되는지 볼 때 | 대상에게 작은 신호를 보내 응답 여부 확인 |
| `traceroute 대상` | 목적지까지 중간 경로를 볼 때 | 패킷이 어떤 네트워크 장비를 거치는지 확인 |
| `whois 대상` | IP나 도메인의 등록 정보를 볼 때 | 소유 기관, 할당 정보 확인 |
| `ip addr` | 내 서버의 네트워크 주소를 볼 때 | 인터페이스별 IP 확인 |
| `nslookup 도메인` | 도메인이 어떤 IP로 해석되는지 볼 때 | DNS 동작 확인 |
| `curl ifconfig.me` | 외부에서 보이는 공인 IP를 확인할 때 | NAT 이후 외부 노출 주소 확인 |

### 5.4 AWS VPC를 손으로 이해하는 흐름

VPC 실습은 버튼을 누르는 순서보다 아래 논리를 잡는 것이 중요하다.

1. VPC CIDR을 정한다.
2. 퍼블릭 서브넷과 프라이빗 서브넷을 나눈다.
3. 퍼블릭 서브넷 라우팅 테이블에 인터넷 게이트웨이를 연결한다.
4. 프라이빗 서브넷은 직접 인터넷 게이트웨이로 보내지 않는다.
5. 프라이빗 서버가 외부로 나가야 하면 NAT 게이트웨이를 사용한다.
6. 보안 그룹에서 필요한 포트만 허용한다.

### 5.5 헷갈렸던 지점

보안 그룹에서 80번 포트를 열어도 라우팅이 잘못되어 있으면 외부에서 접속할 수 없다. 반대로 라우팅이 맞아도 보안 그룹이 막고 있으면 접속할 수 없다.

네트워크 문제는 하나의 설정만 보는 것이 아니라, IP, 라우팅, 게이트웨이, 포트, 보안 규칙을 순서대로 봐야 한다.

### 5.6 나중에 다시 볼 포인트

- 퍼블릭 서브넷과 프라이빗 서브넷의 차이는 라우팅에서 나온다.
- NAT 게이트웨이는 외부에서 내부로 들어오는 통로가 아니다.
- 보안 그룹은 인스턴스 단위, 네트워크 ACL은 서브넷 단위로 이해한다.
- 접속 실패 시 “IP → 라우팅 → 포트 → 애플리케이션” 순서로 확인한다.

## 6. 보충 설명

- 자료 기반 내용: OSI 7계층, IPv4, 퍼블릭/프라이빗 IP, 서브넷, 라우팅, TCP/UDP, 포트, VPC, 인터넷 게이트웨이, NAT 게이트웨이, 보안 그룹, 네트워크 ACL
- 보충 설명: 클라우드 네트워크는 보안 설계와 직접 연결된다. 접속 성공보다 어떤 방향의 트래픽을 허용하는지 이해해야 한다.
- 확실하지 않음: 실습에서 사용한 실제 AWS VPC CIDR 값과 서브넷 CIDR 값은 업로드된 자료만으로는 확실하지 않음.
